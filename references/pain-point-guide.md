## 四、策划洞察提取方法

### 从抱怨中找"机会词"

玩家抱怨 = 需求未被满足的信号：

| 玩家抱怨（负向） | 真实需求（正向） | 设计机会 |
|----------------|---------------|--------|
| "体力不够用" | 想多玩一会儿 | 设计替代体力消耗的趣味循环 |
| "无内容可做" | 想要更多内容 | 设计UGC/常驻活动 |
| "太肝了" | 想轻松玩游戏 | 降低重复劳动，增加休闲玩法 |
| "抽卡太贵" | 想要更多免费奖励 | 设计抽卡保底+免费获取渠道 |
| "匹配不到人" | 想和朋友一起玩 | 降低组队门槛，跨服匹配 |

## 七、常见游戏分析案例

### 二次元抽卡游戏常见痛点
1. 体力限制严格（长草期问题）
2. 抽卡概率不透明
3. 角色强度膨胀（旧角色淘汰）
4. 剧情跳过功能缺失
5. 刷刷刷重复劳动

### 竞技游戏常见痛点
1. 服务器稳定性（卡顿/掉线）
2. 匹配机制（ELO/胜率控制）
3. 外挂问题
4. 平衡性更新（削弱争议）
5. 社交功能缺失

### SLG游戏常见痛点
1. 零氪玩家被碾压
2. 赛季末期鬼服
3. 离线收益设计争议
4. 肝度极高
5. 付费差距过大

## 九、评价时效性筛选方法论（核心）

### 为什么时效性决定一切

- 游戏版本迭代快：1年前的差评可能对应已修复的bug
- 运营问题有时效：某些问题是某个活动专属，活动结束自动消失
- 用户群体变化：用户口味随市场变化，3年前的"氪金吐槽"不等于今天的痛点
- 优先级不同：历史遗留问题 vs 近期新增问题，资源分配策略完全不同

### 黄金标准：90天窗口

| 窗口长度 | 适用场景 | 局限性 |
|---------|---------|---------|
| **≤30天** | 新版本发布后的紧急监测 | 样本量可能不足 |
| **≤90天（推荐）** | 日常竞品监控 | 需要稳定的抓取来源 |
| **≤180天** | 深度分析，覆盖完整更新周期 | 可能混入已修复的历史问题 |
| **>365天** | 仅用于"历史遗留问题"追踪，不用于优先级决策 |

### 版本关联分析方法

1. 提取评论中的版本号（如"1.75版本后出现"）
2. 统计同一版本集中出现的抱怨数量
3. 如果某版本"近期评论+大量差评+同一问题"三者叠加 → **该版本引入的新问题**
4. 如果某问题跨多个版本持续出现 → **长期未解决的结构性问题**，优先级更高

### 差评趋势比绝对值重要

```
近期差评率 = 近90天1-2星评论数 / 近90天总评论数
历史差评率 = 90天前1-2星评论数 / 90天前总评论数

趋势判断：
- 近期 > 历史 → 问题在恶化，需紧急处理
- 近期 ≈ 历史 → 问题持续，需系统性解决
- 近期 < 历史 → 问题在改善，暂缓优先级
```

### 实现注意事项

- 爬取时优先使用平台提供的排序/时间筛选参数（而非先抓全部再过滤）
- 标记评论来源的"更新时间"而非"发布时间"（平台可能编辑旧评论）
- 对于版本号：部分评论不会直接提版本号，可通过评论内容的"更新后…"句式间接判断
